What is Snyk?

Snyk (pronounced "sneak") is a developer-first security platform that automatically finds and fixes vulnerabilities in your open source dependencies, container images, Infrastructure as Code, and application code. It integrates directly into development workflows — IDEs, SCM, CI/CD — rather than sitting outside as a bolt-on tool.

ℹ
Snyk's philosophy: shift security left — catch vulnerabilities at development time, not in production. Developers own the fix, not a separate security team.

// Why Snyk exists

Modern apps are ~80% open source code. Each dependency you pull in (and its transitive dependencies) can carry known CVEs. Without automated scanning, these slip into production undetected. Snyk automates that detection and surfaces actionable remediation paths (upgrade to version X, apply patch Y).

📦

Open Source (SCA)

Scans package manifests (npm, pip, Maven, Go modules, etc.) for vulnerable dependencies.

💻

Code (SAST)

Static analysis of your own code for security bugs — SQL injection, XSS, hardcoded secrets, etc.

🐳

Container

Scans Docker images and base OS layers for CVEs across Alpine, Debian, RHEL, Ubuntu.

☁️

IaC

Finds misconfigurations in Terraform, Helm, Kubernetes YAML, CloudFormation, ARM templates.

Severity Levels & CVSS

Snyk uses the CVSS v3.1 scoring system, augmented with its own exploit intelligence (actual exploit availability, ease of exploitation in real environments).

LevelCVSS ScoreSnyk ActionExample
CRITICAL9.0 – 10.0Block pipeline immediatelyRCE via log4shell (CVE-2021-44228)
HIGH7.0 – 8.9Block or alert depending on policyPrototype pollution with code exec
MEDIUM4.0 – 6.9Warn, create ticketReDoS, SSRF without auth bypass
LOW0.1 – 3.9Track, batch-fixInfo disclosure, timing attacks
💡
Use --severity-threshold=high to only fail builds on HIGH+ findings. This avoids alert fatigue from LOW/MEDIUM noise while still catching what matters.

Dockerized Scanning

Snyk supports two modes for container security: scanning Docker images for OS + app layer CVEs, and running the Snyk CLI itself inside a container for clean CI environments.

// Scanning a Docker image

BASH
# Pull and scan a public image snyk container test nginx:latest # Scan your own built image docker build -t myapp:latest . snyk container test myapp:latest \ --file=Dockerfile \ # include Dockerfile for better context --severity-threshold=high \ --exclude-base-image-vulns # only app-layer vulns, not OS base # Monitor container in Snyk dashboard snyk container monitor myapp:latest --project-name=myapp-prod # Output as JSON for parsing snyk container test myapp:latest --json > container-results.json

// Snyk in a multi-stage Dockerfile

DOCKERFILE
# Stage 1: Build FROM node:20-alpine AS builder WORKDIR /app COPY package*.json ./ RUN npm ci --production COPY . . # Stage 2: Security scan (runs snyk, exits non-zero if vulns found) FROM snyk/snyk:node AS security-check WORKDIR /app COPY --from=builder /app /app ARG SNYK_TOKEN ENV SNYK_TOKEN=$SNYK_TOKEN # Fails build if critical vulns found RUN snyk test --severity-threshold=critical || true # Stage 3: Production image FROM node:20-alpine AS production WORKDIR /app COPY --from=builder /app /app EXPOSE 3000 CMD ["node", "server.js"]

// docker-compose integration

DOCKER-COMPOSE.YML
version: '3.8' services: snyk-scan: image: snyk/snyk:linux volumes: - ./:/project working_dir: /project environment: - SNYK_TOKEN=${SNYK_TOKEN} command: snyk test --severity-threshold=high profiles: [security] # run: docker compose --profile security up

AWS CodeBuild Integration

AWS CodeBuild is the most common Snyk target for teams on AWS. The token is stored in AWS Systems Manager Parameter Store (or Secrets Manager) and injected as an env var.

// Step 1 — Store token in Parameter Store

AWS CLI
aws ssm put-parameter \ --name "/cicd/snyk-token" \ --value "your-snyk-api-token" \ --type SecureString \ --region ap-south-1

// Step 2 — buildspec.yml (full example)

BUILDSPEC.YML
version: 0.2 env: parameter-store: SNYK_TOKEN: /cicd/snyk-token phases: install: runtime-versions: nodejs: 20 commands: - npm install -g snyk - npm ci pre_build: commands: - echo "Running Snyk security scan..." # Fail build on HIGH+ vulnerabilities - snyk test --severity-threshold=high --json > snyk-report.json || BUILD_FAILED=true # Upload snapshot for continuous monitoring - snyk monitor --project-name=my-app-${CODEBUILD_RESOLVED_SOURCE_VERSION} - if [ "$BUILD_FAILED" = "true" ]; then echo "Snyk found HIGH+ vulns"; exit 1; fi build: commands: - npm run build # Container scan after Docker build - docker build -t myapp:$CODEBUILD_RESOLVED_SOURCE_VERSION . - snyk container test myapp:$CODEBUILD_RESOLVED_SOURCE_VERSION --severity-threshold=high post_build: commands: - echo "Security scan complete" reports: snyk-vulnerability-report: files: - snyk-report.json file-format: CUCUMBERJSON artifacts: files: - snyk-report.json
ℹ
Add ssm:GetParameter permission to the CodeBuild IAM role so it can read the Snyk token from Parameter Store.

Jenkins Integration

Store the Snyk token as a Jenkins Credential (Secret Text, ID: snyk-api-token). Use the Snyk Security Scanner plugin OR the CLI directly.

// Declarative Pipeline (Jenkinsfile)

JENKINSFILE (GROOVY)
pipeline { agent { docker { image 'node:20-alpine' } } environment { SNYK_TOKEN = credentials('snyk-api-token') } stages { stage('Install') { steps { sh 'npm ci' sh 'npm install -g snyk' } } stage('Snyk Security Scan') { steps { script { try { sh ''' snyk test \ --severity-threshold=high \ --json > snyk-report.json ''' } catch (err) { // Capture failure but archive report first currentBuild.result = 'UNSTABLE' } } } post { always { archiveArtifacts artifacts: 'snyk-report.json' } } } stage('Snyk Monitor') { when { branch 'main' } steps { sh 'snyk monitor --project-name=my-app' } } stage('Container Scan') { steps { sh 'docker build -t myapp:${BUILD_NUMBER} .' sh 'snyk container test myapp:${BUILD_NUMBER} --severity-threshold=high' } } } post { failure { slackSend channel: '#security', message: "Snyk found vulnerabilities in ${env.JOB_NAME}" } } }

IaC Scanning Deep-Dive

Snyk IaC detects misconfigurations before infrastructure is provisioned. It supports Terraform, CloudFormation, Kubernetes YAML, Helm, Azure ARM, and Ansible.

🏗️

Terraform

.tf files, variables, modules — S3 public buckets, unencrypted RDS, open security groups.

☸️

Kubernetes

privileged containers, missing resource limits, missing network policies, root users.

🌩️

CloudFormation

IAM wildcards, public S3, unencrypted EBS, missing CloudTrail logging.

⛵

Helm

Scans rendered Helm templates — snyk iac test with --helm-chart-path.

BASH — IaC examples
# Terraform — scan plan or .tf files terraform plan -out=tfplan.binary terraform show -json tfplan.binary > tfplan.json snyk iac test tfplan.json # Or scan .tf files directly snyk iac test infrastructure/ # Helm chart scan helm template my-release ./helm-chart > rendered.yaml snyk iac test rendered.yaml # Custom rules (OPA/Rego policies) snyk iac test --rules=./custom-rules.tar.gz

Snyk REST API (Expert Level)

For automation beyond the CLI — bulk reporting, JIRA integration, custom dashboards — use the Snyk REST API v1 (and the newer v3).

BASH — REST API
BASE="https://api.snyk.io/v1" TOKEN="$SNYK_TOKEN" # List all organizations curl -H "Authorization: token $TOKEN" \ $BASE/orgs # List all projects in an org curl -H "Authorization: token $TOKEN" \ $BASE/org/ORG_ID/projects # Get issues for a project curl -H "Authorization: token $TOKEN" \ $BASE/org/ORG_ID/project/PROJECT_ID/issues # Trigger a test via API curl -X POST \ -H "Authorization: token $TOKEN" \ -H "Content-Type: application/json" \ -d '{"packageManager":"npm","encodingType":"plain","projectName":"my-app"}' \ $BASE/test/npm # Create ignore via API (batch ignore from scripts) curl -X POST \ -H "Authorization: token $TOKEN" \ -H "Content-Type: application/json" \ -d '{"ignorePath":"*","reason":"False positive","disregardIfFixable":false}' \ $BASE/org/ORG_ID/project/PROJECT_ID/ignore/ISSUE_ID

Best Practices — Enterprise Setup

🎯

Gate on HIGH+ only

Use --severity-threshold=high in CI. Track LOW/MEDIUM separately. Avoid alert fatigue that leads teams to ignore all security output.

📅

Always set expiry on ignores

Every ignore rule in .snyk MUST have an expires date. This forces periodic review instead of silent permanent suppression.

🔄

snyk monitor on main only

Run snyk test on every branch/PR, but run snyk monitor only on merges to main to keep dashboard clean.

🏷️

Tag projects consistently

Use --tags=team=payments,env=prod to enable filtered reporting and ownership assignment in the Snyk dashboard.

🔐

Rotate service account tokens

Create separate Snyk service accounts per environment (dev/staging/prod). Rotate tokens quarterly via your secrets management system.

📊

Automate JIRA ticket creation

Use Snyk's JIRA integration or the API to auto-create tickets for new HIGH+ findings, assigned to the owning team.
